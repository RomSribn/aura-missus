# PT-0001-004 — Spec

Date: 2026-01-16

## Type
Task (implements PF-0001-001..004)

## Plan

1. Migration: `contact_list(id, owner_id, name, deleted_at, created_at,
   updated_at)` + `contact_list_member(list_id, contact_id)` with unique
   `(list_id, contact_id)` and FKs.
2. Repository + service: create, rename, addMember, removeMember, softDelete.
   All reads filter `deleted_at is null` and scope by `owner_id`.
3. REST: `POST /contact-lists`, `PATCH /contact-lists/{id}`,
   `POST /contact-lists/{id}/members`, `DELETE /contact-lists/{id}/members/{cid}`,
   `DELETE /contact-lists/{id}` (soft).
4. Frontend: Lists page (table), create modal, rename inline, detail drawer with
   member add/remove, delete with confirm.
5. Tests: service-level CRUD + soft-delete-hides-list; one API happy-path test.

## Acceptance
- Create/rename/add/remove/delete all work end to end.
- A deleted list is gone from every list read; its rows still exist in the DB.
- A user cannot see or mutate another user's list.
